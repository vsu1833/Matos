# LSTM Dataset: Current Limitations and Cleaning Recommendations

This document explains the known limitations of the current stock dataset generated by [fetcher.py] and gives practical recommendations for preparing it for LSTM-based `Buy` / `Sell` / `Hold` prediction.

It is written for:
- humans who need a clear preprocessing checklist
- coding copilots that need explicit rules before building training pipelines

## Current Dataset Shape

Each per-ticker CSV in [data/csv] currently contains:

```text
date, close, returns, ema20, ema50, rsi, macd, volatility, volume
```

These columns are intended as model input features:

- `close`
- `returns`
- `ema20`
- `ema50`
- `rsi`
- `macd`
- `volatility`
- `volume`

## Important Limitation: Warmup Rows

Several indicators need historical observations before they can be computed. Because of that, the first rows in each CSV contain missing values.

Example observed in [WIPRO_NS_2018-01-01_2024-12-31.csv]

- `ema20` is missing for the first 18 rows
- `ema50` is missing for the first 48 rows
- `macd` is missing for the first 24 rows
- `volatility` is missing for the first 19 rows

This is expected and not a bug. These features depend on rolling or accumulated history.

### Recommendation

- Do not fill these early missing values with `0`
- Do not fill them using mean or median
- Do not forward-fill them
- Drop all rows where required model features are missing

Practical rule:

```python
feature_cols = [
    "close",
    "returns",
    "ema20",
    "ema50",
    "rsi",
    "macd",
    "volatility",
    "volume",
]

df = df.dropna(subset=feature_cols).reset_index(drop=True)
```

Since `ema50` has the longest warmup among the current features, it usually determines the earliest usable row.

## Limitation: Scale Mismatch Across Features

The feature magnitudes are very different:

- `returns` is typically close to `0`
- `rsi` is roughly in the `0-100` range
- `macd` is usually a small signed number
- `volume` can be in the millions
- `close`, `ema20`, and `ema50` are on price scale

This imbalance can make training unstable or cause the model to overweight large-magnitude columns.

### Recommendation

- Scale features before training
- Fit the scaler only on the training split
- Apply the same fitted scaler to validation and test splits

Recommended options:

- `StandardScaler` for most continuous features
- `log1p(volume)` before scaling if volume spikes are very large

## Limitation: Non-Stationary Price Level

`close`, `ema20`, and `ema50` drift over long periods. This is normal for market data, but it can make it harder for the LSTM to learn stable patterns over multiple years.

### Recommendation

- Keep this in mind during model design and evaluation
- Consider comparing performance with:
  - raw price features
  - normalized price features
  - ratio-style price features such as `close / ema20` and `close / ema50`

Note:
- This is a modeling consideration, not necessarily a required cleaning step
- The current dataset is still usable, but this issue should be monitored during experiments

## Limitation: Outliers

Observed in the current WIPRO dataset:

- `returns` ranges roughly from `-9.2%` to `+16.8%`
- `volume` has large spikes relative to typical values

These may be genuine market events, but they can dominate training if the preprocessing is too naive.

### Recommendation

- Inspect extreme return and volume rows before modifying them
- Avoid removing real market moves unless there is evidence of bad data
- If needed, consider:
  - clipping only extreme tails
  - winsorization
  - log-transforming `volume`

Do not blindly suppress all extremes, because rare but important moves may be informative for `Buy` / `Sell` classification.

## Limitation: Strict Time Ordering Required

LSTMs are sequence models. They assume the data is in chronological order.

### Recommendation

- Ensure rows are always sorted by `date` ascending
- Never shuffle raw rows before creating sequences
- Perform train/validation/test splitting by time, not randomly

Example:

- train: earliest period
- validation: next period
- test: latest period

## Limitation: Sequence Boundaries Across Stocks

Each CSV represents one ticker. If multiple ticker CSVs are later merged into a single training dataset, sequence construction must respect ticker boundaries.

### Recommendation

- Build sequences separately for each ticker
- Concatenate finished samples afterward if needed
- Never allow one LSTM window to contain rows from two different stocks

## Limitation: Label Alignment Risk

If the project predicts `Buy` / `Sell` / `Hold`, the label must come from future movement, not current or past movement.

Typical logic:

```python
forward_return = close.shift(-5) / close - 1
```

Then map:

- `Buy` if future return is above the positive threshold
- `Sell` if future return is below the negative threshold
- `Hold` otherwise

### Recommendation

- Use labels only as training targets
- Never include `label` as an input feature
- Never include `forward_return` as an input feature
- Ensure labels are computed after features, using future prices only

## Limitation: Final Rows in a Labeled Dataset

If labels are based on future returns, the last few rows will not have a valid target because the future window does not exist yet.

For example, with a 5-day forward window:

- the last 5 rows cannot have a valid `forward_return`
- the last 5 rows cannot have a valid `label`

### Recommendation

- Drop rows where the label is missing before supervised training

Example:

```python
df = df.dropna(subset=feature_cols + ["label"]).reset_index(drop=True)
```

## Recommended Cleaning Pipeline

Use the following sequence for each ticker dataset:

1. Load the CSV
2. Parse `date` as datetime
3. Sort by `date`
4. Drop duplicate timestamps if any appear
5. If labels are present, keep them separate from model inputs
6. Drop warmup rows with missing feature values
7. If doing supervised training, drop rows with missing labels
8. Optionally transform `volume` with `log1p`
9. Split by time into train, validation, and test
10. Fit scalers only on training data
11. Apply the same transformations to validation and test data
12. Build rolling LSTM sequences per ticker

## Recommended Rules for Copilots

Any code that prepares this dataset should follow these rules:

- Treat `date` as an ordering column, not a model feature
- Use only these input features unless explicitly changed:
  - `close`
  - `returns`
  - `ema20`
  - `ema50`
  - `rsi`
  - `macd`
  - `volatility`
  - `volume`
- Drop rows with missing feature values instead of imputing indicator warmup gaps
- If labels exist, keep `label` as the target only
- Do not use future-derived columns as model inputs
- Split data chronologically
- Build sequence windows within one ticker at a time
- Fit normalization parameters only on the training split

## Practical Bottom Line

The current dataset is usable, but it is not yet automatically training-ready.

Minimum required cleaning:

- drop warmup rows with missing indicator values
- keep strict date order
- if labels are used, drop rows with missing targets
- scale features before LSTM training
- build sequences separately per ticker

These steps should be treated as required preprocessing, not optional cleanup.
